Continuous Training

Reentrenamiento automático ante drift de datos y concepto

Pipeline & Automatización Intermedio · 3–4 h Nuevo servicio: drift-detector

Objetivos de aprendizaje

Al finalizar esta clase el estudiante será capaz de:

  • Explicar qué es data drift y concept drift, y por qué invalidan modelos en producción.
  • Aplicar los tests KS y Chi-cuadrado para detectar drift de forma estadísticamente rigurosa.
  • Diseñar e implementar un servicio de monitoreo continuo integrado al stack MLOps.
  • Automatizar el ciclo Champion/Challenger: detectar degradación → reentrenar → comparar → promover.
  • Ejecutar un swap de modelo en producción sin downtime usando el endpoint /reload.
  • Registrar métricas de monitoreo en MLFlow para trazabilidad completa del ciclo de vida.
Progresión del tutorial — esta clase añade el componente de monitoreo

Contexto y motivación

El tutorial base construyó un pipeline MLOps funcional: MLFlow registra el modelo, se promueve a Production, y la Inference API lo sirve. Pero ese pipeline asume que el mundo permanece estático. En la práctica, los datos de producción cambian con el tiempo.

Fundamento teórico · Eken et al. (2025)

"Training can be triggered when new data is available or the model performance is reduced or a concept drift is detected." (PS140, PS102, PS147)

"When a change, bias, or drift is captured through monitoring, it often results in adjusting the dataset for model training and/or model optimization." (PS190)

El Continuous Training (CT) es la extensión natural de CI/CD al dominio ML. Mientras en software tradicional el código fuente es el artefacto que puede degradar, en ML los datos son los que degradan el modelo.

🏭 Sin CT (tutorial base)

El modelo se entrena una vez. Se degrada silenciosamente con el tiempo. El equipo se entera cuando el negocio reporta predicciones erróneas.

🔄 Con CT (esta clase)

Un detector monitorea continuamente la distribución de los datos entrantes. Si detecta cambio significativo, dispara reentrenamiento automático.

Tipos de drift

📊 Data Drift (covariante)

La distribución de los features de entrada P(X) cambia respecto al entrenamiento. El modelo no fue entrenado con datos de esta forma.

Ejemplo: Una cámara de sensor envejece y los valores de píxel se desplazan sistemáticamente hacia valores más altos.

🧠 Concept Drift

La relación entre features y etiqueta P(Y|X) cambia. El mismo input ahora debería producir un output diferente.

Ejemplo: Un modelo de detección de fraude entrenado antes de la pandemia; los patrones de compra cambiaron radicalmente.

📉 Performance Drift

Las métricas de negocio (accuracy, recall) caen por debajo de un umbral. Puede ser consecuencia de los dos anteriores.

Detectado via: ground truth con lag (se necesita feedback del mundo real).

📦 Upstream Drift

El pipeline de datos upstream cambió: un schema, un sistema fuente, una lógica de transformación. El modelo recibe datos técnicamente diferentes.

Detectado via: validación de schema + monitoreo de pipelines.

⚠ Esta clase implementa

Data drift (KS test sobre distribución de features) y Concept drift (Chi-cuadrado sobre distribución de predicciones). El performance drift requiere ground truth con lag, que generalmente no está disponible en tiempo real.

Tests estadísticos

Kolmogorov-Smirnov (KS) test — Data drift

Compara dos distribuciones empíricas para determinar si provienen de la misma distribución subyacente. Es no-paramétrico: no asume normalidad.

H₀Las muestras de producción y referencia provienen de la misma distribución continua
EstadísticoD = sup|F₁(x) − F₂(x)| — máxima diferencia entre las funciones de distribución acumuladas
DecisiónSi p-value < α (default 0.05) → rechazar H₀ → drift detectado
VentajaSensible a diferencias en media, varianza Y forma de distribución
LimitaciónRequiere suficientes muestras (mínimo ~20–30 por ventana)
Pythonfrom scipy import stats

# ref_data  = distribución del dataset de entrenamiento (referencia)
# prod_data = ventana de datos recientes de producción

ks_stat, p_value = stats.ks_2samp(ref_data, prod_data)

if p_value < 0.05:
    print(f"DRIFT en feature '{feat}': KS={ks_stat:.4f}, p={p_value:.4f}")

Chi-cuadrado (χ²) — Concept drift

Evalúa si la distribución observada de predicciones difiere significativamente de la esperada según el balance de clases del entrenamiento.

H₀La distribución de predicciones en producción es consistente con la del entrenamiento
Estadísticoχ² = Σ (Oᵢ − Eᵢ)² / Eᵢ
DecisiónSi p-value < α → el modelo "prefiere" ciertas clases → concept drift
InterpretaciónSi el modelo predice siempre clase 2, algo cambió en el espacio de entrada
Pythonfrom scipy import stats
import numpy as np

# observed          = conteo de predicciones por clase en la ventana
# expected_dist     = proporción de clases en el set de entrenamiento

observed = np.bincount(predictions, minlength=3).astype(float)
expected = expected_dist * len(predictions)

chi2_stat, p_value = stats.chisquare(observed, f_exp=expected)

if p_value < 0.05:
    print(f"CONCEPT DRIFT: χ²={chi2_stat:.4f}, p={p_value:.4f}")

El flujo completo de Continuous Training:

Ingesta de producción

El detector ingesta muestras de los datos que llegan a producción (en este tutorial: simuladas con drift progresivo).

Tests estadísticos por ventana

Cada N segundos: KS test sobre cada feature + Chi-cuadrado sobre distribución de predicciones.

Acumulación de evidencia

Solo se dispara el reentrenamiento si drift aparece en K ventanas consecutivas, evitando falsos positivos.

Reentrenamiento automático

El trainer genera un Challenger y lo registra en el Model Registry de MLFlow.

Evaluación Champion/Challenger

Se comparan las métricas de ambas versiones. Se promueve el Challenger solo si supera al Champion.

Zero-downtime swap

POST /reload hace que la Inference API cargue la nueva versión sin reiniciar el contenedor.

Código: detector.py

Clase ProductionBuffer

Simula el flujo de datos de producción. Los primeros 2 ciclos (pasos 0–2) generan datos similares al entrenamiento. A partir del paso 3, introduce drift progresivo en petal length y petal width y sesga las predicciones hacia clase 2.

Diseño pedagógico

La simulación de drift progresivo permite observar en tiempo real cómo el sistema pasa de estado "OK" a "DRIFT detectado" a "REENTRENAMIENTO DISPARADO". En producción, este buffer se reemplaza por la lectura del log de predicciones reales.

Clase DriftDetector

Implementa los dos tests estadísticos y mantiene el contador de ventanas consecutivas:

detector.py · fragmentodef evaluate(self, window_df, predictions) -> dict:
    data_drift    = self.test_data_drift(window_df)    # KS por feature
    concept_drift = self.test_concept_drift(predictions)  # Chi-cuadrado

    any_drift = data_drift["any_feature_drifted"] or concept_drift["drifted"]

    if any_drift:
        self.consecutive_drifts += 1
    else:
        self.consecutive_drifts = 0   # ← reset si hay ventana limpia

    should_retrain = (self.consecutive_drifts >= CONSECUTIVE_DRIFT_WINDOWS)
    ...

El reset del contador ante una ventana sin drift implementa una política conservadora: se requieren ventanas consecutivas, no acumuladas, para evitar falsos positivos.

Código: retrain_trigger.py

Ciclo Champion/Challenger

Este patrón, recomendado por Azure ML best practices (PS153 en Eken et al.), define que el modelo en producción es el Champion. Cada reentrenamiento produce un Challenger que compite contra él.

retrain_trigger.py · fragmentodef trigger(reason, consecutive_windows, mlflow_uri):
    # 1. Registrar el evento de reentrenamiento en MLFlow
    with mlflow.start_run(run_name=f"retrain-event-{reason}"):
        mlflow.log_params({"trigger_reason": reason, ...})
        mlflow.set_tag("event_type", "retrain_trigger")

    # 2. Obtener accuracy del Champion actual
    champion_acc = _get_champion_accuracy(client)

    # 3. Ejecutar el trainer (genera Challenger)
    _run_inline_trainer(mlflow_uri)

    # 4. Comparar versiones
    challenger_version, challenger_acc = _get_challenger_accuracy(client)

    # 5. Decisión: ¿promover o archivar?
    if challenger_acc >= champion_acc + ACCURACY_DELTA:
        client.transition_model_version_stage(
            name=MODEL_NAME, version=challenger_version, stage="Production",
            archive_existing_versions=True
        )
        _reload_inference_api()   # ← zero-downtime swap
    else:
        client.transition_model_version_stage(
            name=MODEL_NAME, version=challenger_version, stage="Archived"
        )
Zero-downtime swap

El endpoint POST /reload de la Inference API carga el nuevo modelo desde el Registry sin reiniciar el contenedor. Mientras el nuevo modelo carga, el anterior sigue sirviendo predicciones. Es el mecanismo que habilita el CT sin interrupciones de servicio.

Ejercicios

Ejercicio 1

Observar el ciclo completo de CT

Levanta el stack y observa el comportamiento del detector en los logs.

  1. Ejecuta docker compose logs -f drift-detector
  2. Observa los pasos 0, 1, 2 (sin drift) — los tests deben reportar ✓ OK
  3. En el paso 3, el drift comienza — observa cómo cambian los p-values de KS
  4. En el paso 4 (segunda ventana con drift), observa el mensaje 🔁 RETRAINING TRIGGERED
  5. Abre MLFlow UI (http://localhost:5001) y verifica los experimentos:
    • iris-drift-monitoring — métricas de detección por ventana
    • iris-retraining-events — registros de triggers y promociones
    • iris-production — el nuevo run de entrenamiento con tag triggered_by=drift_detector
Ejercicio 2

Ajustar sensibilidad del detector

Modifica los parámetros en docker-compose.yml y observa el efecto:

  1. Cambia KS_P_VALUE_THRESHOLD a 0.10 — ¿detecta drift antes o después?
  2. Cambia CONSECUTIVE_DRIFT_WINDOWS a 3 — ¿cuánto tarda más en disparar el reentrenamiento?
  3. Reduce WINDOW_SIZE a 15 — ¿qué efecto tiene en la sensibilidad del test KS?
  4. Discute: ¿cuál es el trade-off entre sensibilidad (detectar drift rápido) y especificidad (evitar falsos positivos)?
Ejercicio 3 · Avanzado

Implementar drift real desde predicciones de la API

En lugar de datos simulados, modifica el ProductionBuffer para leer las predicciones reales:

  1. Agrega un endpoint GET /predictions/log a app.py que retorne las últimas N predicciones con sus inputs
  2. Modifica ProductionBuffer.ingest_production_sample() para llamar a ese endpoint via httpx
  3. Verifica que el detector funciona con datos reales haciendo llamadas con features desplazadas:
    curl -X POST http://localhost:8000/predict -d '{"instances": [[7.5, 4.0, 6.5, 2.5]]}'
  4. Observa si el KS test detecta el shift en la distribución de petal length
Ejercicio 4 · Investigación

Comparar métodos de detección de drift

El KS test es uno de varios detectores posibles. Investiga e implementa una alternativa:

  1. Reemplaza el KS test por el test de Population Stability Index (PSI) para data drift
  2. Implementa el algoritmo ADWIN (Adaptive Windowing) del paquete river para detección online
  3. Compara ambos métodos en términos de: velocidad de detección, tasa de falsos positivos, costo computacional
  4. Registra los resultados comparativos como métricas en MLFlow en un experimento dedicado

Preguntas de discusión

  1. ¿Cuándo NO conviene automatizar el reentrenamiento? Eken et al. (2025) identifica la "Automation Decision Dilemma" como un desafío real. ¿Qué actividades requieren supervisión humana en el ciclo CT?
  2. El detector usa distribución de predicciones para concept drift, no ground truth. ¿Qué limitaciones tiene este enfoque? ¿Cuándo puede producir falsos positivos?
  3. ¿Qué ocurre si el Challenger nunca supera al Champion? Diseña una política para este escenario — ¿cuántos intentos fallidos antes de escalar a intervención humana?
  4. El reentrenamiento usa el mismo dataset Iris. En un sistema real, ¿de dónde vendrían los nuevos datos de entrenamiento? Discute el rol del feedback loop de producción.
