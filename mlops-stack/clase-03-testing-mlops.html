Testing en MLOps

Quality Gates: tests de datos, modelo y API que bloquean la promoción

Pipeline & Automatización Intermedio · 3–4 h Nuevo servicio: test-runner

Objetivos de aprendizaje

  • Distinguir los tres niveles de testing en MLOps: datos, modelo y API.
  • Implementar quality gates que bloquean automáticamente la promoción de modelos deficientes.
  • Escribir tests estadísticos de integridad de datos con pytest.
  • Validar el contrato de una Inference API con tests de integración reales (sin mocks).
  • Registrar resultados de QA como métricas en MLFlow para trazabilidad.
  • Configurar el test-runner como servicio Docker con profiles.

La pirámide de tests en MLOps

El testing en software tradicional sigue la pirámide clásica (muchos unit tests, pocos E2E). En MLOps, la pirámide tiene una dimensión adicional: los datos son un artefacto que también debe testearse.

API Tests test_api.py
Model Tests + Quality Gates test_model.py
Data Tests test_data.py

Más tests → base más amplia → ejecución más frecuente

📦 Nivel 1 · Data Tests

Corren antes del entrenamiento. Validan schema, completitud, rangos, balance de clases. Si fallan, el pipeline no debe proceder.

🧠 Nivel 2 · Model Tests

Corren después del entrenamiento, antes del deploy. Implementan quality gates: accuracy, F1, robustez, sanity checks contra baseline.

🌐 Nivel 3 · API Tests

Corren contra la API en producción. Validan contrato, latencia SLA, manejo de errores. Sin mocks — contra el servicio real.

📊 MLFlow Integration

Todos los resultados se registran como métricas en el experimento quality-assurance. La decisión de promoción queda completamente trazada.

Fundamento teórico · Eken et al. (2025)

"Data quality involves assessing the scientific quality of data with domain experts, checking data completeness, checking consistency in data format and structure." (PS62, PS24, PS151)

"Model quality involves assessing the model's predictive performance through performance metrics and test sets (PS21), validating model bias and fairness (PS100), monitoring model to detect overfitting issues (PS59)." (Sección 3.2.8)

Quality Gates: el concepto clave

Un quality gate es una condición binaria que debe cumplirse para que el pipeline continúe. Si falla, la promoción se bloquea automáticamente — sin intervención humana.

GateUmbral defaultConsecuencia si falla
Accuracy mínima0.90Modelo no se promueve a Production
F1-score weighted0.90Modelo no se promueve a Production
Recall por clase0.80Modelo no se promueve a Production
Latencia P50500msAPI falla los acceptance tests
Data completeness0% nulosPipeline no inicia entrenamiento

Todos los umbrales son configurables vía variables de entorno en docker-compose, sin necesidad de modificar el código.

⚠ Automation Decision Dilemma

Eken et al. (2025) identifica este dilema explícitamente: "model validation processes require communication and approval from regulatory agencies, fairness and bias identification and analysis, root cause analysis from monitoring alerts." No todo debe automatizarse. Los quality gates automáticos funcionan para umbrales técnicos objetivos; decisiones de negocio o regulatorias requieren revisión humana.

Ejemplo representativo

test_data.pydef test_class_balance(self, iris_data):
    """
    Las clases deben estar razonablemente balanceadas.
    Si una clase tiene menos del 10% → posible sesgo severo.
    """
    _, y, _, _ = iris_data
    counts = pd.Series(y).value_counts()
    for cls, count in counts.items():
        pct = count / len(y)
        assert pct >= 0.10, (
            f"Clase {cls} tiene solo {pct:.1%} — posible clase minoritaria"
        )

El sanity check más importante

test_model.pydef test_better_than_random_baseline(self, model, iris_splits):
    """
    El modelo DEBE superar un clasificador aleatorio por al menos 20%.
    Si no lo supera, algo salió muy mal en el entrenamiento.
    """
    X_train, X_test, y_train, y_test, _ = iris_splits
    baseline = DummyClassifier(strategy="stratified", random_state=42)
    baseline.fit(X_train, y_train)

    baseline_acc = accuracy_score(y_test, baseline.predict(X_test))
    model_acc    = accuracy_score(y_test, model.predict(X_test))

    assert model_acc >= baseline_acc + 0.20, (
        f"El modelo ({model_acc:.4f}) no supera al baseline "
        f"({baseline_acc:.4f}) por el margen mínimo de 0.20"
    )
Quality Gate inline en train.py

Además de los tests en el contenedor separado, train.py incluye una función run_quality_gate() que evalúa accuracy y F1 inmediatamente después del entrenamiento. Si falla, el modelo no se registra en el Registry. Es la primera línea de defensa, antes de que el test-runner entre en acción.

Nivel 3: API Tests

Archivo: tests/test_api.py

Sin mocks

Estos tests llaman a la Inference API real en la red Docker. No usan mocks ni stubs. Esto verifica que el sistema funciona de extremo a extremo, incluyendo la serialización JSON, el routing de FastAPI y la carga del modelo desde MLFlow.

Categorías

ClaseTestsQué valida

TestPredictContract11Schema completo de request/response, consistencia class_id/name, predicciones correctas de setosa y virginica
TestLatency3Single prediction SLA, promedio de 10 requests, batch latency
TestErrorHandling43 features → 422, 5 features → 422, lista vacía → 4xx, strings → 422

Test de latencia con SLA

test_api.pydef test_average_latency_over_10_requests(self, api_url, max_latency_ms):
    """La latencia promedio en 10 requests no debe exceder el SLA."""
    latencies = []
    for _ in range(10):
        _, _, lat = http_post(f"{api_url}/predict", VALID_SINGLE)
        latencies.append(lat)
    avg = sum(latencies) / len(latencies)
    assert avg < max_latency_ms, (
        f"Latencia promedio {avg:.1f}ms excede el SLA de {max_latency_ms}ms\n"
        f"Latencias: {[f'{l:.0f}' for l in latencies]}"
    )

Orquestador: run_tests.py

Corre los tres niveles en orden, registra todo en MLFlow y termina con exit code 0 (OK) o 1 (falló). El exit code es lo que CI/CD usa para decidir si bloquear el pipeline.

run_tests.py · flujo# Nivel 1 → Data Tests
r1 = run_pytest("test_data.py")
if not r1["ok"]: all_ok = False   # ← bloquea pipeline

# Nivel 2 → Model Tests (quality gates configurables)
r2 = run_pytest("test_model.py", args=["--min-accuracy=0.90", ...])
if not r2["ok"]: all_ok = False   # ← modelo no se promueve

# Nivel 3 → API Tests (solo si la API está disponible)
r3 = run_pytest("test_api.py", args=["--max-latency=500"])
if not r3["ok"]: all_ok = False

# Registrar todo en MLFlow experiment "quality-assurance"
log_results_to_mlflow(results, all_ok)

sys.exit(0 if all_ok else 1)  # ← CI/CD lee este código

Configuración docker-compose

El test-runner usa Docker Compose profiles para no ejecutarse automáticamente con docker compose up. Solo corre cuando se invoca explícitamente.

docker-compose.yml · fragmentotest-runner:
    build: ./tests
    environment:
      GATE_MIN_ACCURACY:   "0.90"   # ← configurable sin tocar código
      GATE_MIN_F1:         "0.90"
      GATE_MAX_LATENCY_MS: "500"
    profiles:
      - testing             # ← solo corre con --profile testing

Cómo ejecutar la suite

Asegurarse de que el stack está corriendo

docker compose up -d mlflow inference-api (el modelo debe estar entrenado y en Production)

Construir el test-runner

docker compose --profile testing build test-runner

Ejecutar la suite completa

docker compose --profile testing run --rm test-runner

Ver resultados en MLFlow

Abrir http://localhost:5001 → Experiment quality-assurance

Correr solo un nivel específico

docker compose --profile testing run --rm test-runner python -m pytest test_data.py -v

Ejercicios

Ejercicio 1

Hacer fallar un quality gate deliberadamente

  1. En docker-compose.yml, cambia GATE_MIN_ACCURACY a "0.99"
  2. Ejecuta el test-runner: docker compose --profile testing run --rm test-runner
  3. Observa el mensaje QUALITY GATE FAILED y el exit code 1
  4. Verifica en MLFlow que el run de quality-assurance registra gates_passed=0
  5. Restaura el umbral a 0.90 y vuelve a correr — debe pasar
Ejercicio 2

Agregar un test de sesgo por clase

Agrega a test_model.py un test que verifique que ninguna clase tiene más del 60% de las predicciones (detección de modelo sesgado):

  1. Crea el método test_no_class_dominates_predictions en TestSanityChecks
  2. Calcula la frecuencia de cada clase en y_pred
  3. Aserta que ninguna clase supera el 60% de las predicciones totales
  4. Verifica que el test pasa con el modelo actual
  5. Modifica el modelo para que siempre prediga clase 0 y comprueba que el test falla
Ejercicio 3

Integrar los tests con el Drift Detector

Modifica el retrain_trigger.py (Clase 01) para que ejecute el test-runner como quality gate antes de promover el challenger:

  1. En retrain_trigger.py, después de _train_challenger(), agrega una llamada a los model tests
  2. Si los tests fallan, archivar el challenger en lugar de promoverlo
  3. Registrar el resultado en MLFlow con tag qa_gate_passed
  4. Verificar el flujo completo: drift detectado → reentrenamiento → QA → promoción o archivo
Ejercicio 4 · Investigación

Implementar mutation testing

El mutation testing verifica que tus tests son suficientemente robustos para detectar bugs. Usa mutmut para evaluar la calidad de la suite:

  1. Instala mutmut en el Dockerfile del test-runner
  2. Ejecuta mutmut run --paths-to-mutate=../model-trainer/train.py
  3. Analiza qué mutaciones sobreviven (no son detectadas por los tests)
  4. Agrega tests adicionales hasta que el mutation score supere el 80%

Preguntas de discusión

  1. ¿Por qué los API tests no usan mocks? En testing tradicional, los mocks son una práctica recomendada. ¿En qué casos un test con mock puede dar falsa confianza en un sistema MLOps?
  2. Los data tests corren sobre el dataset Iris estático. En un sistema real donde los datos vienen de un data lake que cambia constantemente, ¿cuándo y con qué frecuencia deberían correr los data tests?
  3. ¿Cuál es el quality gate más importante? Si el equipo debe priorizar un solo test por restricciones de tiempo de CI/CD, ¿cuál elegirían y por qué?
  4. El test de latencia usa time.perf_counter(). ¿Qué factores pueden hacer que este test sea inestable en un entorno CI/CD compartido? ¿Cómo se mide latencia de forma más robusta en producción?
