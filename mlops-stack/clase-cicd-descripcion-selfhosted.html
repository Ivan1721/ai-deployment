CI/CD con GitHub Actions
Self-Hosted Runner

Automatización completa del pipeline MLOps: build, test y deploy corren en la misma máquina donde vive el stack, sin infraestructura externa ni credenciales de red.

Pipeline & Automatización Avanzado · 4–5 h 3 workflows · 1 script de setup

Objetivos de aprendizaje

  • Comprender qué es un self-hosted runner y por qué elimina la complejidad de red frente a un runner en la nube.
  • Explicar la diferencia entre CI (Integración Continua) y CD (Entrega Continua) y sus responsabilidades en MLOps.
  • Construir un docker-compose.yml que distinga servicios permanentes (con restart) de servicios efímeros (jobs de CI/CD).
  • Diseñar un workflow de CI que bloquee un PR si el modelo no supera los quality gates de la Clase 03.
  • Diseñar un workflow de CD con rolling update (zero-downtime) y verificación post-deploy.
  • Conectar el reentrenamiento programado con el modelo existente en MLFlow, reutilizando el ciclo de la Clase 01.
  • Identificar qué decisiones de diseño son específicas de MLOps vs. las que vienen del DevOps tradicional.

Concepto clave: ¿por qué self-hosted?

Un pipeline CI/CD convencional usa runners en la nube del proveedor (en GitHub, runs-on: ubuntu-latest). Ese runner recibe el código, corre los jobs, y cuando necesita desplegar, hace SSH a un servidor remoto. Para un stack MLOps esto introduce varios problemas:

  • El test-runner necesita llegar a MLFlow y a la Inference API — que están en otra máquina — para correr los quality gates. Esto requiere exponer puertos, configurar IPs, manejar credenciales.
  • Las imágenes Docker se construyen en el runner (nube), luego se publican en un registry (GHCR), y el servidor las descarga. Tres pasos donde antes había uno.
  • El deploy requiere SSH, claves, IPs, rutas remotas — cuatro Secrets que pueden rotar o fallar.

Con un self-hosted runner el proceso es el runner es la máquina de producción. Cuando el job corre docker compose up -d, lo hace directamente sobre el stack que está sirviendo tráfico. No hay red que cruzar.

Stieler y Bauer proponen un Git workflow donde "a CI/CD tool hosted on a resource-rich machine executes ML pipelines for training, testing, and integrating ML models based on the code committed by developers." El self-hosted runner implementa exactamente ese patrón.

Construcción del Docker Compose

Cómo se diseña el docker-compose.yml para coexistir con el runner de CI/CD en la misma máquina.

Red y volumen compartido

La primera decisión de diseño es definir una red con nombre propio y un volumen persistente. Esto es lo que permite que el runner, el test-runner y los servicios de producción se comuniquen sin configuración adicional.

docker-compose.ymlsin cambios
networks: mlops-net: driver: bridge red con nombre → DNS interno entre contenedores volumes: mlflow-data: persiste entre deploys y reboots del host

La red mlops-net es el punto clave: cuando el runner ejecuta docker compose run test-runner, ese contenedor se une automáticamente a mlops-net. Desde ahí, http://mlflow:5001 resuelve por DNS interno — el mismo mecanismo que usan los servicios entre sí. Así desaparece el problema de red que tenía la versión anterior.

El volumen mlflow-data es el otro elemento crítico: almacena la base de datos SQLite de MLFlow y todos los artefactos del modelo. Al sobrevivir entre deploys, el modelo entrenado en una ejecución anterior sigue disponible en la siguiente — el CD puede detectar si ya existe y saltar el entrenamiento.

Servicios permanentes: los cuatro pilares

Cuatro servicios corren de forma continua en producción. Todos comparten el mismo patrón de construcción:

docker-compose.yml — patrón de servicio permanentesin cambios en lógica
inference-api: build: ./inference-api construye localmente siempre container_name: inference-api nombre fijo → referenciable desde scripts restart: unless-stopped NUEVO: reinicia si cae, no si se detiene manualmente ports: - "8000:8000" expuesto al host → accesible desde el runner environment: MLFLOW_TRACKING_URI: http://mlflow:5001 DNS interno MODEL_NAME: iris-classifier MODEL_STAGE: Production volumes: - mlflow-data:/mlflow acceso al volumen compartido depends_on: mlflow: condition: service_healthy no arranca hasta que MLFlow pase health check healthcheck: test: ["CMD", "curl", "-sf", "http://localhost:8000/health"] interval: 10s retries: 20 start_period: 60s 60s de gracia para cargar el modelo desde MLFlow networks: - mlops-net

Cada uno de los cuatro servicios permanentes (mlflow, inference-api, drift-detector, nginx) sigue este mismo patrón. Las diferencias entre ellos son: qué puertos exponen, qué variables de entorno necesitan, y si tienen healthcheck propio.

Servicios efímeros: jobs de CI/CD

model-trainer y test-runner son fundamentalmente diferentes: no deben correr de forma continua. Se ejecutan una vez, hacen su trabajo, y terminan. Por eso no tienen restart.

docker-compose.yml — servicios efímerossin cambios
model-trainer: build: ./model-trainer container_name: model-trainer # Sin restart: → Docker no lo reinicia automáticamente # Se invoca con: docker compose run --rm model-trainer # El flag --rm elimina el contenedor al terminar environment: MLFLOW_TRACKING_URI: http://mlflow:5001 MLFLOW_ARTIFACT_ROOT: /mlflow/artifacts volumes: - mlflow-data:/mlflow escribe el modelo en el volumen compartido depends_on: mlflow: condition: service_healthy networks: - mlops-net test-runner: build: ./tests container_name: test-runner # Sin restart: → tampoco se reinicia # Se invoca con: docker compose run --rm test-runner environment: MLFLOW_TRACKING_URI: http://mlflow:5001 INFERENCE_API_URL: http://inference-api:8000 DNS interno — funciona en CI GATE_MIN_ACCURACY: "0.90" GATE_MIN_F1: "0.90" GATE_MAX_LATENCY_MS: "500" volumes: - mlflow-data:/mlflow networks: - mlops-net

La clave del diseño: restart: unless-stopped

Esta política es lo que hace que el stack sea autónomo en la misma máquina del runner:

PolíticaComportamientoCuándo usarla
no (default)No reinicia nuncaJobs CI/CD, one-shot tasks
alwaysReinicia siempre, incluso si se detuvo manualmenteServicios críticos sin mantenimiento
unless-stoppedReinicia si cae, pero respeta docker compose stopServicios de producción con CI/CD
on-failureSolo si termina con errorJobs que pueden fallar legítimamente

unless-stopped es la política correcta aquí porque el CD necesita poder hacer docker compose stop inference-api para el rolling update sin que Docker lo reinicie inmediatamente. Con always, el contenedor volvería a arrancar antes de que el CD pueda reemplazarlo.

Construcción del Workflow CI

Archivo: .github/workflows/ci.yml — se dispara en cada Pull Request.

Estructura base: trigger y runs-on

La primera sección de cualquier workflow define cuándo corre y dónde. Son las dos decisiones más importantes del archivo.

ci.yml — sección on: y runs-on:modificado
name: CI — Build & Test on: pull_request: branches: [main, develop] corre en cualquier PR que apunte a main o develop push: branches: [develop] también corre en push directo a develop # El trigger pull_request es el que bloquea el merge: # si el job falla, GitHub no permite merge hasta que pase. jobs: lint: runs-on: self-hosted CAMBIO: antes era ubuntu-latest # self-hosted le dice a GitHub que busque un runner registrado # con ese label en nuestro repositorio — no una VM en la nube.
¿Por qué dos jobs separados (lint y test)?
Separar lint de test permite que el lint falle rápido (en segundos) sin gastar tiempo levantando Docker. En CI, la velocidad de feedback importa: el desarrollador sabe si hay un error de sintaxis en 30 segundos, no en 5 minutos. El job test tiene needs: lint — no corre si lint falla.

Job lint: validación estática rápida

El primer job solo necesita Python — no Docker, no servicios. Valida que el código tiene la sintaxis correcta y que el docker-compose.yml es válido antes de intentar construir nada.

ci.yml — job lint completológica sin cambios
lint: name: Lint runs-on: self-hosted steps: - uses: actions/checkout@v4 descarga el código del PR - name: Lint Python run: | pip install flake8 --quiet flake8 model-trainer/train.py \ inference-api/app.py \ drift-detector/detector.py \ tests/run_tests.py tests/test_*.py \ --max-line-length=110 \ --extend-ignore=E501,W503 \ --statistics || true || true → reporta pero no bloquea el PR por estilo - name: Validate docker-compose run: docker compose config --quiet detecta errores de YAML antes de usarlo

Step: build images

Construye todas las imágenes Docker en paralelo usando la cache local del host. Como el runner vive en la misma máquina que el stack de producción, las capas ya construidas en deploys anteriores están en cache — el build es rápido.

ci.yml — step buildsin GHCR
- name: Build images run: docker compose build --parallel # --parallel: construye mlflow, model-trainer, inference-api, # drift-detector, tests, nginx simultáneamente. # A diferencia de la versión anterior (runner en la nube), # aquí no hay login a GHCR ni push de imágenes. # Las imágenes quedan en el daemon Docker local del host.

Step: start MLFlow

MLFlow debe estar corriendo antes de que el trainer pueda registrar el modelo. Este step lo levanta y espera explícitamente al health check en lugar de simplemente dormir N segundos. Si MLFlow no responde en 120 segundos, el step falla con un mensaje claro.

ci.yml — step MLFlow
- name: Start MLFlow run: | docker compose up -d mlflow levanta en background # Espera activa: consulta el estado del health check # cada 3 segundos hasta que aparezca "(healthy)" timeout 120 bash -c \ 'until docker compose ps mlflow | grep -q "(healthy)"; do sleep 3; done' echo "MLFlow healthy ✓" # El health check está definido en docker-compose.yml: # test: ["CMD", "curl", "-sf", "http://localhost:5001/"] # Docker marca el contenedor como healthy cuando curl retorna 0.

Step: train model

Ejecuta el trainer como contenedor efímero. --rm elimina el contenedor al terminar — el modelo queda registrado en MLFlow (en el volumen compartido), no en el contenedor.

ci.yml — step train
- name: Train model run: docker compose run --rm model-trainer # docker compose run → inicia un contenedor one-shot # --rm → lo elimina cuando termina # model-trainer → servicio definido en docker-compose.yml # El trainer hace: # 1. Conectarse a http://mlflow:5001 (DNS interno) # 2. Entrenar RandomForest sobre Iris # 3. Registrar métricas, parámetros y el modelo en MLFlow # 4. Promover la nueva versión a stage "Production" # 5. Salir con código 0 (éxito) o 1 (fallo) # Si sale con 1, el step falla y el PR queda bloqueado.

Step: start inference-api

Levanta la API y espera hasta que responda al health check. La start_period de 60 segundos que está en el docker-compose.yml da tiempo para que la API cargue el modelo desde el Registry de MLFlow antes de que Docker la evalúe.

ci.yml — step inference-api
- name: Start inference-api run: | docker compose up -d inference-api # Espera activa sobre el puerto del host (no DNS interno) # porque aquí estamos en el runner, no en un contenedor timeout 120 bash -c \ 'until curl -sf http://localhost:8000/health > /dev/null; do sleep 4; done' echo "inference-api ready ✓"
¿Por qué localhost aquí pero mlflow:5001 en otros pasos?
Dentro del proceso del runner (bash), usamos localhost:8000 — el puerto expuesto al host. Dentro de un contenedor Docker (como test-runner), usamos http://inference-api:8000 — el nombre DNS interno de la red. Son dos perspectivas distintas de la misma máquina.

Step: QA suite — el quality gate

Este es el step más importante del CI. Ejecuta la misma suite de tests de la Clase 03. Si algún quality gate falla, el exit code del contenedor es 1, el step falla, el job falla, y GitHub bloquea el merge del PR.

ci.yml — step QA (el corazón del CI)
- name: Run QA suite run: | docker compose run --rm \ -e MLFLOW_TRACKING_URI=http://mlflow:5001 \ DNS interno — funciona porque comparte mlops-net -e INFERENCE_API_URL=http://inference-api:8000 \ test-runner # El test-runner corre los tres niveles de la Clase 03: # Nivel 1: Data Tests (21 tests sobre el dataset) # Nivel 2: Model Tests + Quality Gates (accuracy ≥ 0.90, F1 ≥ 0.90) # Nivel 3: API Tests (contrato, latencia SLA, manejo de errores) # Exit code 0 → todos los gates pasaron → CI verde → PR puede mergearse # Exit code 1 → algún gate falló → CI rojo → PR bloqueado # Por qué funciona el DNS interno aquí: # docker compose run agrega test-runner a mlops-net. # mlflow e inference-api están en mlops-net. # El resolver DNS de Docker resuelve "mlflow" a su IP en la red.

Step: cleanup

El cleanup corre siempre — incluso si los tests fallaron — gracias a if: always(). Elimina los contenedores creados durante el CI pero no los volúmenes: el modelo entrenado en MLFlow persiste para el próximo run.

ci.yml — step cleanup
- name: Cleanup CI containers if: always() corre incluso si los pasos anteriores fallaron run: docker compose down --remove-orphans # docker compose down → detiene y elimina los contenedores # --remove-orphans → elimina contenedores huérfanos # Sin -v → conserva mlflow-data entre runs de CI # Si se agregara -v, el modelo entrenado se borraría y habría # que reentrenar en cada run de CI — costoso e innecesario.

Construcción del Workflow CD

Archivo: .github/workflows/cd.yml — se dispara solo en push a main.

Trigger y control de concurrencia

El CD tiene dos propiedades que no tiene el CI: se dispara solo en main (no en PRs), y tiene un bloque de concurrencia para evitar que dos deploys corran simultáneamente.

cd.yml — sección on: y concurrency:nuevo
on: push: branches: [main] solo en merge a main — no en PRs env: DEPLOY_DIR: /opt/mlops-stack directorio permanente en el host COMPOSE_PROJECT: mlops nombre del proyecto para docker compose -p jobs: deploy: runs-on: self-hosted concurrency: group: production-deploy nombre del grupo de concurrencia cancel-in-progress: false espera en cola — no cancela el deploy anterior # cancel-in-progress: false es crucial en CD: # si dos merges llegan en rápida sucesión, el segundo # espera a que el primero termine en lugar de interrumpirlo. # Interrumpir un deploy a mitad puede dejar el stack en # estado inconsistente.

Step: sync files — rsync en lugar de SSH

El runner hace checkout del repositorio en su directorio de trabajo temporal. Necesitamos copiar esos archivos al directorio permanente /opt/mlops-stack donde vive el stack. rsync sincroniza solo los archivos que cambiaron.

cd.yml — step rsyncreemplaza SSH
- name: Sync files to deploy directory run: | sudo mkdir -p /opt/mlops-stack sudo rsync -av --delete \ --exclude='.git/' \ no copiar historial git --exclude='__pycache__/' \ no copiar cachés de Python --exclude='*.pyc' \ ./ /opt/mlops-stack/ # rsync -av --delete: # -a → archive: preserva permisos, timestamps, symlinks # -v → verbose: muestra qué archivos cambiaron # --delete → elimina en destino los archivos que ya no están en origen # (garantiza que el directorio de deploy refleja exactamente # el estado del repositorio en ese commit)

Step: build — reconstruir con el código nuevo

Construye las imágenes desde el directorio de deploy (ya actualizado con rsync). Como el daemon Docker tiene cache de las capas anteriores, solo se reconstruyen las capas que cambiaron.

cd.yml — step build
- name: Build updated images working-directory: /opt/mlops-stack trabajar desde el directorio permanente run: | docker compose -p mlops build --parallel # -p mlops: nombre del proyecto. # Evita que los contenedores de CI (que también usan # docker compose) interfieran con los de producción. # Los contenedores de producción se llaman mlops_mlflow_1, # los de CI se llaman con el nombre del directorio de trabajo.

Step: ensure MLFlow is running

MLFlow puede estar ya corriendo de un deploy anterior (gracias a restart: unless-stopped). Este step lo levanta si no está, o lo deja como está si ya responde. Es idempotente.

cd.yml — step ensure MLFlow
- name: Ensure MLFlow is running working-directory: /opt/mlops-stack run: | docker compose -p mlops up -d mlflow up -d es idempotente: no recrea si ya corre timeout 120 bash -c \ 'until docker compose -p mlops ps mlflow | grep -q "(healthy)"; do sleep 3; done' echo "MLFlow healthy ✓"

Step: train model if needed

Esta es la lógica más sofisticada del CD: antes de entrenar, consulta el Model Registry de MLFlow para saber si ya existe un modelo en Production. Si existe (volumen persistente de un deploy anterior), no reentrenar — ese modelo sigue siendo válido. Si no existe (primer deploy o volumen eliminado), entrenar.

cd.yml — step train if needednuevo
- name: Train model if needed working-directory: /opt/mlops-stack run: | # Consultar el Registry directamente desde Python dentro del contenedor MODEL_EXISTS=$(docker compose -p mlops run --rm \ -e MLFLOW_TRACKING_URI=http://mlflow:5001 \ model-trainer python -c " import mlflow, os mlflow.set_tracking_uri(os.environ.get('MLFLOW_TRACKING_URI')) from mlflow import MlflowClient c = MlflowClient() try: vs = c.get_latest_versions('iris-classifier', stages=['Production']) print('exists' if vs else 'missing') except Exception: print('missing') " 2>/dev/null || echo "missing") if echo "$MODEL_EXISTS" | grep -q "missing"; then echo "No model in Production — training..." docker compose -p mlops run --rm model-trainer else echo "Model already in Production — skipping training" reutiliza modelo existente fi

Step: rolling update (zero-downtime)

En lugar de docker compose down && docker compose up (que causa downtime), el CD actualiza un servicio a la vez. La inference-api se actualiza primero y se verifica antes de continuar con los demás.

cd.yml — rolling updatenuevo
- name: Update inference-api (zero-downtime) working-directory: /opt/mlops-stack run: | docker compose -p mlops up -d \ --no-deps \ no reiniciar dependencias (mlflow) --build \ usar la imagen recién construida inference-api sleep 8 dar tiempo para que el nuevo contenedor cargue el modelo # Verificar que la API responde correctamente curl -sf http://localhost:8000/health > /dev/null \ && echo "inference-api healthy ✓" \ || (echo "UNHEALTHY — rollback needed" && exit 1) - name: Update remaining services run: | docker compose -p mlops up -d --no-deps drift-detector nginx # mlflow no se actualiza aquí — tiene su propio ciclo de vida # y restart: unless-stopped garantiza que sigue corriendo
¿Qué pasa si la verification falla?
El step termina con exit 1, el job CD falla, y GitHub Actions notifica al equipo. El contenedor anterior sigue corriendo porque Docker no lo eliminó — up -d --no-deps reemplaza el contenedor, pero si el nuevo falla el healthcheck, el stack queda con el contenedor nuevo (no el viejo) en estado degradado. Para un rollback manual: git revert + push a main dispara un nuevo CD con el código anterior.

Step: post-deploy QA

La misma suite de tests de la Clase 03 corre de nuevo, ahora contra el stack en producción real. Detecta cualquier problema de integración que no apareció durante el CI.

cd.yml — post-deploy QA
- name: Post-deploy QA working-directory: /opt/mlops-stack run: | docker compose -p mlops run --rm \ -e MLFLOW_TRACKING_URI=http://mlflow:5001 \ -e INFERENCE_API_URL=http://inference-api:8000 \ test-runner # Esta es la diferencia entre CI y CD: # En CI: los tests validan el candidato (¿merece llegar a producción?) # En CD: los tests validan producción (¿el deploy fue exitoso?) # Si falla aquí, el deploy se completó pero algo salió mal. # Hay que investigar — el código ya está en main.

Step: deploy summary

Escribe un resumen en el panel de GitHub Actions. Este resumen es visible en la interfaz web de GitHub para cualquier miembro del equipo, sin necesidad de leer los logs completos.

cd.yml — summary
- name: Deploy summary if: always() corre incluso si algo falló run: | # $GITHUB_STEP_SUMMARY es una variable especial de GitHub Actions # Todo lo que se escribe ahí aparece como Markdown en la UI echo "## Deploy Summary" >> $GITHUB_STEP_SUMMARY echo "- **Commit:** \`${{ github.sha }}\`" >> $GITHUB_STEP_SUMMARY echo "- **Status:** ${{ job.status }}" >> $GITHUB_STEP_SUMMARY docker compose -p mlops ps \ --format "table {{.Name}}\t{{.Status}}" \ >> $GITHUB_STEP_SUMMARY estado de cada contenedor en la tabla

Construcción del Workflow de Reentrenamiento

Archivo: .github/workflows/retrain.yml — complementa el drift-detector de la Clase 01.

Trigger: schedule + workflow_dispatch

Este workflow tiene dos formas de dispararse: automáticamente en un horario fijo (cron), o manualmente desde la UI de GitHub Actions con parámetros configurables. La combinación de ambos da flexibilidad total.

retrain.yml — triggersnuevo
on: schedule: - cron: "0 3 * * 1" lunes 03:00 UTC — cuando el tráfico es bajo workflow_dispatch: habilita el botón "Run workflow" en la UI de GitHub inputs: min_accuracy: description: "Accuracy mínima del quality gate" default: "0.90" min_f1: description: "F1-score mínimo" default: "0.90" # workflow_dispatch permite disparar el reentrenamiento: # - Desde la UI: GitHub → Actions → Scheduled Retrain → Run workflow # - Desde la API: curl -X POST .../actions/workflows/retrain.yml/dispatches # - Con parámetros distintos: ej. min_accuracy=0.95 para un release crítico

Steps del reentrenamiento

retrain.yml — steps completos
steps: # 1. Verificar que el stack está corriendo antes de intentar reentrenar - name: Verify stack is running run: | docker compose -p mlops \ -f /opt/mlops-stack/docker-compose.yml \ usar el compose de producción ps mlflow | grep -q "(healthy)" # Si MLFlow no está healthy, el step falla aquí con mensaje claro # 2. Entrenar el nuevo modelo (challenger) - name: Retrain model working-directory: /opt/mlops-stack run: | docker compose -p mlops run --rm model-trainer # Registra el nuevo modelo como una nueva versión en el Registry # y lo promueve a Production (archivando la versión anterior) # 3. Correr quality gates sobre el nuevo modelo - name: Run QA on new model run: | docker compose -p mlops run --rm \ -e GATE_MIN_ACCURACY=${{ github.event.inputs.min_accuracy || '0.90' }} \ -e GATE_MIN_F1=${{ github.event.inputs.min_f1 || '0.90' }} \ test-runner # || '0.90': fallback para cuando se dispara por cron (sin inputs) # 4. Hot-reload del modelo en la API sin reiniciar el contenedor - name: Hot-reload model in inference-api if: success() solo si QA pasó — si falló, no recargar run: | curl -sf -X POST http://localhost:8000/reload # Este endpoint (definido en app.py desde el Tutorial Base) # carga la nueva versión del modelo desde MLFlow sin # reiniciar el proceso de uvicorn — zero-downtime.
Relación con la Clase 01 (drift-detector)
El drift-detector reacciona a cambios estadísticos en los datos en tiempo real — puede disparar reentrenamiento en cualquier momento del día. El retrain.yml garantiza un ciclo de actualización periódico incluso si el drift no es severo: por ejemplo, al actualizar dependencias del modelo, cambiar hiperparámetros, o incorporar correcciones al dataset. Los dos mecanismos son complementarios.

Quality gates antes de promote PS111 (AWS), PS62 El test-runner bloquea el merge si accuracy o F1 no alcanzan los umbrales mínimos

Ejercicios

Ejercicio 1 · Observar el CI en acción

  1. Crear una rama: git checkout -b test/ci-demo
  2. Cambiar el umbral de accuracy en docker-compose.yml: GATE_MIN_ACCURACY: "0.99"
  3. Hacer commit y push, abrir un PR hacia develop
  4. Observar en GitHub → Actions cómo el CI corre y el QA suite falla
  5. Revertir el cambio, hacer push — el CI debe pasar y el PR habilitarse

Ejercicio 2 · Trigger manual de reentrenamiento

  1. Ir a GitHub → Actions → Scheduled Retrain → Run workflow
  2. Cambiar min_accuracy a 0.95
  3. Observar si el modelo existente pasa el gate o no
  4. Verificar en MLFlow que aparece un nuevo run con tag triggered_by=cicd
  5. Hacer curl http://localhost:8000/health y verificar que model_version cambió

Ejercicio 3 · Investigación: Automation Decision Dilemma

Eken et al. (2025) identifican el "Automation Decision Dilemma" como un desafío real en MLOps: no todo debe automatizarse. Con el pipeline implementado, discutir:

  1. ¿En qué casos el post-deploy QA automático no es suficiente? (ej: modelos de diagnóstico médico)
  2. ¿Cómo agregarías un paso de aprobación humana en el CD antes de actualizar inference-api? (hint: GitHub Environments con required reviewers)
  3. ¿Qué métricas de negocio — más allá de accuracy y F1 — deberían evaluarse antes de un deploy en producción?
