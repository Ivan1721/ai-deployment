CI/CD con GitHub Actions

Integración y entrega continua para el pipeline MLOps. Automatiza build, test y deploy en cada commit.

3 archivos nuevos 1 archivo modificado Todo lo anterior se conserva

¿Qué cambia respecto a la versión anterior?

La versión anterior del stack requería intervención manual para construir imágenes, ejecutar tests y desplegar. Esta versión agrega automatización completa mediante GitHub Actions, manteniendo exactamente la misma lógica de todos los componentes.

ci.yml — Workflow de Integración Continua

Cuándo corre: en cada Pull Request hacia main o develop, y en push directo a develop.

Qué hace: garantiza que el código es correcto antes de hacer merge. Si falla, el PR queda bloqueado.

Job 1: lint

ci.yml- name: Lint Python (flake8)
  run: |
    flake8 model-trainer/train.py inference-api/app.py ...
      --max-line-length=110
  continue-on-error: true   # reporta pero no bloquea por estilo

Job 2: build (con cache en GHCR)

ci.yml- name: Build inference-api
  uses: docker/build-push-action@v5
  with:
    context:    ./inference-api
    push:       false    # en CI no publicamos, solo compilamos
    cache-from: type=registry,ref=ghcr.io/ORG/inference-api:cache
    cache-to:   type=registry,ref=ghcr.io/ORG/inference-api:cache,mode=max

El cache en GHCR hace que los builds subsecuentes sean mucho más rápidos — solo se reconstruyen las capas que cambiaron.

Job 3: test (QA suite)

ci.yml# Levantar MLFlow + entrenar + correr test-runner
- name: Run QA suite
  run: |
    docker compose run --rm \
      -e MLFLOW_TRACKING_URI=http://mlflow:5001 \
      -e INFERENCE_API_URL=http://inference-api:8000 \
      test-runner
# exit code 1 → quality gate falló → CI falla → PR bloqueado
En CI, el DNS interno funciona
En el runner de GitHub Actions todos los contenedores están en la misma red Docker, así que mlflow e inference-api se resuelven por nombre sin el workaround de host-gateway. El extra_hosts del docker-compose solo actúa en desarrollo local.

cd.yml — Workflow de Entrega Continua

Cuándo corre: solo en push a main (merge de PR aprobado y con CI verde).

Qué hace: construye y publica todas las imágenes en GHCR, luego despliega en el servidor de producción vía SSH.

Tag de imagen = SHA del commit

cd.yml- name: Generate image tag
  id: tag
  run: echo "tag=sha-${GITHUB_SHA::8}" >> $GITHUB_OUTPUT

# Cada imagen se publica con dos tags:
#   sha-a1b2c3d4  → inmutable, trazable al commit exacto
#   latest        → conveniente para docker compose pull
tags: |
  ghcr.io/ORG/inference-api:sha-a1b2c3d4
  ghcr.io/ORG/inference-api:latest

Deploy en el servidor (rolling update)

cd.yml — SSH al servidor# Actualizar inference-api primero (zero-downtime)
docker compose up -d --no-deps inference-api
sleep 10
curl -sf http://localhost:8000/health || \
  (echo "unhealthy after update" && exit 1)

# Actualizar resto de servicios
docker compose up -d --no-deps mlflow drift-detector nginx

# Post-deploy: correr suite QA completa
docker compose run --rm test-runner

Environment de producción (aprobación manual opcional)

cd.ymldeploy:
  environment: production
  # Si configuras "Required reviewers" en el environment "production"
  # en GitHub → Settings → Environments, cada deploy requerirá
  # aprobación manual de un reviewer antes de ejecutarse.

retrain.yml — Reentrenamiento Programado

Cuándo corre: todos los lunes a las 03:00 UTC, o manualmente desde la UI de GitHub Actions.

Disparo manual con parámetros

retrain.ymlworkflow_dispatch:
  inputs:
    min_accuracy:
      description: "Accuracy mínima del quality gate"
      default:     "0.90"
    force_promote:
      description: "Promover aunque no supere al champion"
      default:     "false"

Para dispararlo manualmente: GitHub → Actions → Scheduled Retrain → Run workflow.

Complemento al drift-detector (Clase 01)
El drift-detector reacciona a cambios estadísticos en los datos en tiempo real. El retrain.yml garantiza reentrenamiento periódico aunque el drift no sea severo — por ejemplo, al cambiar los hiperparámetros del modelo o actualizar dependencias.

Paso 1: Configurar el repositorio GitHub

Crear repositorio en GitHub (público o privado). El Container Registry (GHCR) funciona con ambos.

Subir el código:
git init && git add . && git commit -m "initial: mlops stack with ci/cd"
git remote add origin https://github.com/TU_ORG/TU_REPO.git
git push -u origin main

Habilitar GHCR para el repositorio: GitHub → Settings → Actions → General → "Allow GitHub Actions to create and approve pull requests" ✓

Crear rama develop: git checkout -b develop && git push -u origin develop. Los PRs van de feature branches → develop → main.

Proteger la rama main: GitHub → Settings → Branches → Add rule → require status checks (CI workflow) before merging.

Paso 2: Preparar el servidor de producción

El servidor puede ser cualquier VPS con Ubuntu 22.04+ (AWS EC2, DigitalOcean, etc.) con al menos 2 CPU y 4GB RAM.

Ejecutar el script de setup en el servidor:
bash setup_server.sh
Instala Docker, Docker Compose, crea el directorio de deploy y el servicio systemd.

Generar clave SSH para GitHub Actions (en tu máquina local):
ssh-keygen -t ed25519 -C "github-actions" -f ~/.ssh/mlops_deploy -N ""

Agregar la clave pública al servidor:
cat ~/.ssh/mlops_deploy.pub | ssh usuario@servidor "cat >> ~/.ssh/authorized_keys"

Editar el archivo .env en el servidor (/opt/mlops-stack/.env):
REGISTRY_IMAGE=ghcr.io/TU_ORG
IMAGE_TAG=latest

Primer deploy manual para verificar:
cd /opt/mlops-stack && docker compose up -d

Paso 3: Configurar Secrets en GitHub

GitHub → Settings → Secrets and variables → Actions → New repository secret:

PROD_HOSTIP o hostname del servidor. Ej: 203.0.113.10
PROD_USERUsuario SSH del servidor. Ej: ubuntu o deploy
PROD_SSH_KEYContenido completo del archivo ~/.ssh/mlops_deploy (clave privada)
PROD_DEPLOY_PATHRuta del directorio en el servidor. Ej: /opt/mlops-stack
⚠ GITHUB_TOKEN es automático
El secret GITHUB_TOKEN lo provee GitHub Actions automáticamente — no necesitas crearlo. Se usa para autenticarse en GHCR y para publicar artefactos del workflow.

Configurar el environment de producción (aprobación manual)

Para requerir aprobación humana antes de cada deploy: GitHub → Settings → Environments → New environment → production → Required reviewers → agregar tu usuario.

GitHub Container Registry (GHCR)

Las imágenes se publican automáticamente en ghcr.io/TU_ORG/. Para verlas: GitHub → Packages.

Visibilidad de paquetes

Por defecto los paquetes son privados. Para hacerlos públicos (necesario si el servidor hace pull sin autenticación):
GitHub → Packages → nombre-imagen → Package settings → Change visibility → Public.

Pull manual desde el servidor

# Autenticarse (si las imágenes son privadas)
echo TOKEN | docker login ghcr.io -u USUARIO --password-stdin

# Verificar que las imágenes están disponibles
docker pull ghcr.io/TU_ORG/inference-api:latest

Cambios detallados en docker-compose.yml

El único cambio funcional al archivo existente es agregar el campo image: a cada servicio. Antes:

ANTES (docker-compose.yml)inference-api:
  build: ./inference-api
  container_name: inference-api
  ...

Después:

DESPUÉS — campo image: agregadoinference-api:
  image: ${REGISTRY_IMAGE:-mlops}/inference-api:${IMAGE_TAG:-latest}
  build: ./inference-api   # build sigue funcionando localmente
  container_name: inference-api
  ...

Con image: presente, Docker Compose puede hacer pull de la imagen publicada en GHCR en lugar de construirla localmente. En desarrollo local, si REGISTRY_IMAGE no está definida, usa el fallback mlops/inference-api:latest y sigue construyendo normalmente con docker compose build.

Flujo de trabajo diario con Git

# 1. Crear rama para nueva funcionalidad
git checkout -b feature/nueva-feature

# 2. Hacer cambios y commit
git add .
git commit -m "feat: descripción del cambio"

# 3. Push y abrir PR hacia develop
git push origin feature/nueva-feature
# → GitHub Actions dispara ci.yml automáticamente
# → Si CI pasa: PR puede aprobarse y mergearse
# → Si CI falla: PR queda bloqueado hasta corregir

# 4. Merge develop → main (cuando se quiere ir a producción)
# → GitHub Actions dispara cd.yml automáticamente
# → Build + push de imágenes a GHCR
# → Deploy al servidor vía SSH
# → Post-deploy QA suite

# 5. Reentrenamiento manual bajo demanda
# → GitHub → Actions → Scheduled Retrain → Run workflow
✓ Convención de commits recomendada
