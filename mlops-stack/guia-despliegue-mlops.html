Requisitos previos

verificar versiones
$ docker --version # Docker version 24.0+ $ docker compose version # Docker Compose version v2.20+ $ docker info | grep -i memory # Total Memory: al menos 4GB

Estructura del proyecto

árbol de directorios
mlops-stack/ ├── docker-compose.yml # orquestación de todos los servicios ├── start.sh # script de inicio ordenado │ ├── mlflow/ # Tracking Server + Model Registry ├── model-trainer/ # entrenamiento y registro del modelo ├── inference-api/ # FastAPI que sirve predicciones ├── nginx/ # reverse proxy │ ├── drift-detector/ # Clase 01: detección de drift └── tests/ # Clase 03: suite de QA

Build inicial (una sola vez)

Construir todas las imágenes antes del primer uso. Toma ~5–10 min dependiendo de la conexión.

bash
# Construir todas las imágenes en paralelo $ docker compose build --parallel # O construir una imagen específica $ docker compose build mlflow $ docker compose build inference-api $ docker compose build drift-detector $ docker compose build test-runner # Forzar rebuild sin cache (si hubo cambios en el código) $ docker compose build --no-cache test-runner

Reset y limpieza

⚠ Importante
Correr down -v elimina los volúmenes con la base de datos de MLFlow y todos los modelos registrados. El modelo deberá entrenarse nuevamente.
bash
# Detener todos los contenedores (conserva datos) $ docker compose down # Detener Y eliminar volúmenes (reset completo) $ docker compose down -v # Eliminar también las imágenes construidas $ docker compose down -v --rmi local # Ver qué está corriendo en este momento $ docker compose ps
📦 Tutorial Base

Levantar el stack base

Servicios incluidos: MLFlow, Inference API, Nginx.

Opción A — Script automático (recomendado)

bash
$ bash start.sh

El script respeta el orden correcto: espera el health check de MLFlow → entrena el modelo → levanta la API → verifica que responde.

Opción B — Paso a paso (para entender el flujo)

bash
# 1. Levantar MLFlow primero $ docker compose up -d mlflow # 2. Esperar hasta que esté healthy (~15s) $ docker compose ps # mlflow-server running (healthy) # 3. Entrenar y registrar el modelo $ docker compose run --rm model-trainer # 4. Levantar la API y el proxy $ docker compose up -d inference-api nginx # 5. Verificar que la API cargó el modelo $ docker compose logs --tail=10 inference-api

Reentrenar el modelo manualmente

bash
# Entrenar nueva versión (la promueve a Production automáticamente) $ docker compose run --rm model-trainer # Recargar el modelo en la API sin reiniciar el contenedor $ curl -X POST http://localhost:8000/reload

Detener el stack base

bash
# Detener sin borrar datos (el modelo sigue registrado en MLFlow) $ docker compose down # La próxima vez que levantes, el modelo ya está en el Registry # No necesitas correr model-trainer de nuevo $ docker compose up -d mlflow inference-api nginx
🔄 Clase 01 · Continuous Training — Drift Detection

Levantar stack con drift detector

El drift detector se agrega al stack base. Requiere que el modelo ya esté entrenado y en Production.

bash — primera vez
# 1. Limpiar estado anterior $ docker compose down -v # 2. Construir imagen del detector $ docker compose build drift-detector # 3. Levantar MLFlow $ docker compose up -d mlflow # 4. Entrenar y registrar el modelo (esperar healthy primero) $ docker compose run --rm model-trainer # 5. Levantar API e iniciar detector $ docker compose up -d inference-api drift-detector nginx
bash — si el stack base ya está corriendo
# Agregar solo el detector al stack activo $ docker compose up -d drift-detector

Observar los ciclos de detección

bash
# Seguir logs del detector en tiempo real $ docker compose logs -f drift-detector # Ver solo los últimos 50 logs $ docker compose logs --tail=50 drift-detector

Comportamiento esperado en los logs:

output esperado
# Ciclos 0-2: sin drift --- Cycle 0 (drift_phase=False) --- Status: ✓ OK Data drift: False Consecutive: 0/2 # Ciclo 3: drift comienza --- Cycle 3 (drift_phase=True) --- Status: ⚠ DRIFT DETECTED Data drift: True features=['petal length (cm)', 'petal width (cm)'] Consecutive: 1/2 # Ciclo 4: reentrenamiento disparado --- Cycle 4 (drift_phase=True) --- 🔁 RETRAINING TRIGGERED (consecutive=2) ✅ Promoting Challenger v2 → Production

Ajustar parámetros del detector

Todos los parámetros son variables de entorno en docker-compose.yml. Cambiarlos y reiniciar el detector:

VariableDefaultDescripción
KS_P_VALUE_THRESHOLD"0.05"Umbral KS: p-value bajo el cual se detecta data drift por feature
CHI2_P_VALUE_THRESHOLD"0.05"Umbral Chi²: p-value para concept drift sobre distribución de predicciones
CONSECUTIVE_DRIFT_WINDOWS"2"Ventanas consecutivas con drift antes de disparar reentrenamiento
WINDOW_SIZE"30"Muestras por ventana de evaluación estadística
CHECK_INTERVAL_S"30"Segundos entre ciclos de detección
ejemplo: detector más sensible
# Editar docker-compose.yml y cambiar los valores, luego: $ docker compose stop drift-detector $ docker compose up -d drift-detector # O sobreescribir en línea sin editar el archivo $ docker compose run --rm \ -e KS_P_VALUE_THRESHOLD=0.10 \ -e CONSECUTIVE_DRIFT_WINDOWS=1 \ -e CHECK_INTERVAL_S=10 \ drift-detector

Verificar resultados en MLFlow

Drift metricshttp://localhost:5001Experiment: drift-monitoring
Retrain eventshttp://localhost:5001Experiment: retraining-events
Model versionshttp://localhost:5001/#/models/iris-classifierRegistry
verificar vía API
# Ver qué versión está en Production actualmente $ curl http://localhost:8000/health | python3 -m json.tool # Forzar recarga del modelo (tras promoción manual) $ curl -X POST http://localhost:8000/reload
✓ Qué buscar en MLFlow
En el experiment drift-monitoring: cada run corresponde a una ventana de detección. Las métricas ks_pval_* bajan gradualmente hasta cruzar el umbral 0.05. En retraining-events: los runs con tag triggered_by=drift_detector muestran el historial de reentrenamientos automáticos.
🧪 Clase 03 · Testing en MLOps — Quality Gates

Build del test-runner

bash
# Construir imagen del test-runner (necesario antes del primer uso) $ docker compose build test-runner # Si hubo cambios en los archivos de test, forzar rebuild $ docker compose build --no-cache test-runner
⚠ Requiere stack base activo
El test-runner necesita que MLFlow esté corriendo (para cargar el modelo del Registry). La inference-api es opcional — si no está disponible, los tests de Nivel 3 se saltan automáticamente.

Correr la suite completa

bash — suite completa (3 niveles)
# Asegurarse de que MLFlow está corriendo $ docker compose up -d mlflow inference-api # Correr la suite completa $ docker compose run --rm test-runner

Output esperado al finalizar:

output esperado
╔══════════════════════════════════════════════════════╗ ║ QA SUITE — RESULTS SUMMARY ║ ╠══════════════════════════════════════════════════════╣ ║ data ✓ PASS (21 passed, 0 failed, 1.2s) ║ ║ model ✓ PASS (13 passed, 0 failed, 3.4s) ║ ║ api ✓ PASS (18 passed, 0 failed, 2.1s) ║ ╠══════════════════════════════════════════════════════╣ ║ ✓ PROMOTION ALLOWED ║ ╚══════════════════════════════════════════════════════╝

El exit code del contenedor indica el resultado:

bash
$ docker compose run --rm test-runner; echo "Exit: $?" # Exit: 0 → todos los quality gates pasaron # Exit: 1 → al menos un gate falló → promoción bloqueada

Correr por nivel individual

Nivel 1 — Data Tests

bash
$ docker compose run --rm test-runner \ python -m pytest test_data.py -v # Ver solo los tests fallidos $ docker compose run --rm test-runner \ python -m pytest test_data.py -v --tb=short -q

Nivel 2 — Model Tests + Quality Gates

bash
$ docker compose run --rm \ -e MLFLOW_URI=http://172.18.0.1:5001 \ -e MIN_ACCURACY=0.90 \ -e MIN_F1=0.90 \ test-runner python -m pytest test_model.py -v # Correr solo los quality gates (sin sanity checks ni robustez) $ docker compose run --rm test-runner \ python -m pytest test_model.py::TestPerformanceGates -v

Nivel 3 — API Tests

bash
$ docker compose run --rm \ -e API_URL=http://172.18.0.1:8000 \ -e MAX_LATENCY=500 \ test-runner python -m pytest test_api.py -v # Solo tests de latencia $ docker compose run --rm test-runner \ python -m pytest test_api.py::TestLatency -v # Solo tests de manejo de errores $ docker compose run --rm test-runner \ python -m pytest test_api.py::TestErrorHandling -v

Ajustar quality gates

VariableDefaultDescripción
GATE_MIN_ACCURACY"0.90"Accuracy mínima en el test set para aprobar el gate
GATE_MIN_F1"0.90"F1-score weighted mínimo aceptable
GATE_MAX_LATENCY_MS"500"Latencia máxima en ms para el SLA de la API
ejemplos de configuración
# Ejercicio: hacer fallar el gate de accuracy $ docker compose run --rm \ -e GATE_MIN_ACCURACY=0.99 \ test-runner # Gate más estricto para latencia (200ms SLA) $ docker compose run --rm \ -e GATE_MAX_LATENCY_MS=200 \ test-runner # Gate mínimo (para exploración) $ docker compose run --rm \ -e GATE_MIN_ACCURACY=0.50 \ -e GATE_MIN_F1=0.50 \ test-runner

Resultados en MLFlow

QA resultshttp://localhost:5001Experiment: quality-assurance
✓ Qué buscar en MLFlow
Cada ejecución del test-runner genera un run en el experiment quality-assurance. Las métricas data_ok, model_ok, api_ok y gates_passed permiten rastrear el historial de QA de cada versión del modelo. El tag promoted=True/False indica si el modelo fue autorizado para producción.

Para demostrar el flujo completo: el detector genera drift → dispara reentrenamiento → el test-runner valida el challenger antes de promoverlo.

bash — levantar todo

Esperar health check (~15s) $ docker compose ps # 5. Entrenar modelo inicial $ docker compose run --rm model-trainer # 6. Levantar todos los servicios $ docker compose up -d inference-api drift-detector nginx # 7. Verificar que todo está corriendo $ docker compose ps # 8. Correr la suite de QA sobre el modelo inicial $ docker compose run --rm test-runner # 9. Seguir los logs del detector en otra terminal $ docker compose logs -f drift-detector

Troubleshooting

MLFlow no arranca (unhealthy)

diagnóstico
$ docker compose logs mlflow # Si dice "unable to open database file" → problema de permisos en volumen # Solución: reset de volúmenes $ docker compose down -v $ docker compose up -d mlflow

inference-api no carga el modelo

diagnóstico
$ docker compose logs inference-api # Si dice "degraded" → el modelo no está en Production en el Registry # Solución: entrenar primero $ docker compose run --rm model-trainer # Verificar que hay un modelo en Production $ curl http://localhost:5001/api/2.0/mlflow/registered-models/get \ --data-urlencode "name=iris-classifier"

test-runner no alcanza inference-api

diagnóstico
# Ver qué IP detectó el entrypoint $ docker compose run --rm test-runner cat /proc/net/route # La gateway es la IP del host en la red Docker bridge # Si el entrypoint detectó la IP incorrecta, sobreescribir manualmente # Primero encontrar la IP correcta: $ docker network inspect mlops-stack_mlops-net | grep Gateway # Luego correr con esa IP (ejemplo: 172.18.0.1) $ docker compose run --rm \ -e MLFLOW_TRACKING_URI=http://172.18.0.1:5001 \ -e INFERENCE_API_URL=http://172.18.0.1:8000 \ test-runner

drift-detector no genera experimentos en MLFlow

diagnóstico
$ docker compose logs drift-detector # Buscar líneas con "MLFlow logging failed" o "IMPORT ERROR" # Si el contenedor no aparece corriendo: $ docker compose ps drift-detector # Restart del detector $ docker compose restart drift-detector $ docker compose logs -f drift-detector

Puerto 5001 o 8000 ocupado

diagnóstico y solución
# Ver qué proceso usa el puerto $ lsof -i TCP:5001 $ lsof -i TCP:8000 # En macOS: el puerto 5000 lo usa AirPlay — por eso usamos 5001 # Si 5001 también está ocupado, cambiar en docker-compose.yml: # ports: "5002:5001" (host:container)

Comandos de diagnóstico generales

bash
# Estado de todos los contenedores $ docker compose ps # Logs de todos los servicios simultáneamente $ docker compose logs -f --tail=20 # Logs de un servicio específico $ docker compose logs -f mlflow $ docker compose logs -f inference-api $ docker compose logs -f drift-detector # Entrar a un contenedor para inspeccionar $ docker compose exec mlflow bash $ docker compose exec inference-api bash # Ver uso de recursos $ docker stats
